## Context

Today's `pyproject.toml` line 38 floor is `aiosmtplib>=3.0.2`. The uv
resolver already lands on `aiosmtplib==5.1.0` — but a fresh install on
another machine could legally resolve to a 3.x version, where SMTPUTF8 /
8BITMIME negotiation differs.

This task locks the floor to match what the test suite (TASK-006) and the
retrofitted providers (TASK-002/003/004) actually run against. Resolves
brainstorm Q5 — pinned **permissive** at `>=5.0`, not strict `>=5.1`.

Implements Spec §3 Module 5.

## Implementation Notes

### Pattern to Follow

```bash
# 1. Activate venv (project rule — always activate before uv/python).
source .venv/bin/activate

# 2. Edit pyproject.toml. From the project root:
#    Change line 38:
#       "aiosmtplib>=3.0.2",
#    to:
#       "aiosmtplib>=5.0",

# 3. Regenerate the lock.
uv lock

# 4. Verify the lock resolves cleanly against the locked file (no resolver
#    surprises).
uv sync --frozen --extra all --extra dev

# 5. Spot-check the resolved version.
uv pip show aiosmtplib | head -2
# Expected: Version: 5.1.0 (or a 5.x patch newer than 5.0)
```

### Key Constraints

- **Do not pin `==5.1.0` or `>=5.1`** — brainstorm Q5 resolved this as
  permissive `>=5.0`. The strict pin would force fresh installs to
  upgrade through a patch even when consumer environments are happy on 5.0.
- **Do not add `aiosmtplib` to any other optional-dependencies group**;
  it's a core runtime dep.
- **Test independence**: this task has no code dependency on TASK-001/
  002/003/004 and can be executed in any order. In a sequential worktree
  flow, completing it early de-risks lock conflicts on the eventual hotfix
  branch.
- Verify `uv.lock` is tracked by git before adding it: `git ls-files
  --error-unmatch uv.lock`. If the command errors, `uv.lock` is gitignored
  in this repo and only `pyproject.toml` should be committed.

- `notify/providers/mail.py:16` — `import aiosmtplib` (just the import).
- `notify/providers/mail.py:66-104` — the `connect()` body that uses the
